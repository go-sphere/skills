Capture scaffold conventions so generated proposals stay consistent with expected go-sphere scaffold style.

## Scope

This document is the source of truth for scaffold conventions used by this skill.
Rules in this document are AI generation-time constraints and output self-check expectations.
Do not depend on scripts or automation to enforce them.

## Proto File Modes

For proto files generated or modified by this skill in the current task:

1. `service proto`: contains at least one `service` declaration.
2. `message-only proto`: contains messages/enums only and no `service` declaration.

Mode-aware policy:

1. `service proto` must pass single-service topology checks and service-local error placement checks.
2. `message-only proto` is allowed and exempt from service-only topology checks.
3. `message-only proto` still must pass naming/import/runtime/codegen compatibility checks.

## Package and Domain Boundaries

### `api.v1`

- Public API domain.
- Uses HTTP annotations.
- Reuses `shared.v1.User` for user-facing identity payloads.
- Paths are mostly `/api/...` with one `/v1/auth/wxmini` exception.

### `dash.v1`

- Admin dashboard domain.
- Strongly follows action-style paths:
  - `/api/<resource>/list`
  - `/api/<resource>/create`
  - `/api/<resource>/update`
  - `/api/<resource>/detail/{id}`
  - `/api/<resource>/delete/{id}`
- Reuses `entpb.*` messages heavily for CRUD payloads.

### `shared.v1`

- Shared services and common messages.
- Contains reusable messages (`User`) and shared service APIs (`StorageService`).
- Includes `test.proto` demonstrating advanced binding patterns.

### `bot.v1`

- Non-HTTP integration with `sphere.options.options` metadata.
- Used for route generation keyed by bot command metadata.

### `entpb`

- Generated by `entproto`.
- Do not edit manually.
- Treat as storage-oriented reusable message set.

## HTTP Path and Method Style

1. Do not force pure REST resource style if module already uses action-style endpoints.
2. In `dash.v1`, keep existing action paths and method choices (including `POST` for update).
3. In `api.v1`, follow existing per-service path prefix choices (`/api` or `/v1`) for compatibility.

## Service-Level Route Namespace Rule

Use one stable route prefix per proto service to avoid cross-service collisions.

Examples from scaffold style:

1. `AdminService` routes start with `/api/admin...`.
2. `AdminSessionService` routes start with `/api/admin-session...`.
3. `KeyValueStoreService` routes start with `/api/key-value-store...`.

When designing a new service, define the service prefix first, then define all service RPC paths under that prefix.

## Pagination and List Response Style

Scaffold default list style (especially in `dash.v1`):

- Request: `page`, `page_size` (int64)
- Response: `repeated <items>`, `total_size`, `total_page`
- Request message often includes:
  - `option (sphere.binding.default_location) = BINDING_LOCATION_QUERY;`

Prefer this style for scaffold-consistent proposals unless a feature explicitly requires cursor pagination.

## Binding and Validation Patterns

1. Use `google.api.http` on all HTTP-exposed RPCs.
2. Use `BINDING_LOCATION_URI` for URI path IDs.
3. Use message-level query default for list/filter request messages when appropriate.
4. Use `buf.validate` for required/range rules.
5. Use CEL validation for embedded complex objects (e.g., `entpb.Admin` constraints in `dash/v1/admin.proto`).
6. Advanced patterns available and valid in this repo:
  - `response_body`
  - `default_auto_tags`
  - explicit field binding tags

## Error Enum Patterns

Observed scaffold conventions:

1. Service-specific domain enums are defined in the same proto file as the service.
2. Always include `*_UNSPECIFIED = 0`.
3. Usually set `option (sphere.errors.default_status) = 500`.
4. Use `(sphere.errors.options)` for per-value `status` and `message`.
5. `reason` is optional and used only when needed.
6. For `service proto`, keep one service-local business error enum prefix aligned with the service/file prefix when feasible.

Common/shared errors can live in `shared.v1` when multiple services depend on the same semantics.

## Message Reuse Strategy (Scaffold-First)

Use this priority:

1. Reuse `entpb` in admin/internal CRUD APIs when contract allows.
2. Reuse `shared.v1` for cross-service/common messages.
3. Add custom DTO/VO only when contract shaping requires it.

## Naming and Compatibility Notes

1. Default style is snake_case field names.
2. Existing compatibility-driven exceptions exist (e.g., `dash/v1/auth.proto` uses camelCase response fields like `accessToken`), and lint rules are configured accordingly.
3. Do not â€śnormalizeâ€ť compatibility fields if clients depend on them.
4. For `service proto`, enforce strict file-prefix mapping:
   - File prefix uses snake_case.
   - Service and service-local error prefixes use PascalCase.
   - Example: `admin_session.proto` -> `AdminSessionService` and `AdminSessionError`.

## Proto Topology Rules for Skill Outputs

For `service proto` files generated by this skill:

1. Exactly one service per `.proto` file.
2. Top-level declaration order: `service` -> `message` -> `error enum`.
3. If a required topology rule fails, the output is non-deliverable and must include `Validation Notes -> Blocking Issues` with a corrected proposal.

## Codegen and Lint Signals From Scaffold

1. `buf.gen.yaml` uses managed mode with `go_package_prefix` override.
2. Plugin chain includes `protoc-gen-go`, `protoc-gen-sphere-errors`, `protoc-gen-sphere`, `protoc-gen-route`.
3. `protoc-gen-sphere-binding` is configured via `buf.binding.yaml`.
4. Lint ignores include generated `proto/entpb` and the special-case `proto/dash/v1/auth.proto`.

Use these signals to avoid proposing patterns that conflict with actual generation/runtime behavior.

## Route Conflict Guardrails for Scaffold

1. Current runtime commonly uses Gin-compatible routing behavior.
2. Keep wildcard names stable for same resource branches (`:id` vs `:task_id` divergence can panic in Gin trees).
3. Avoid introducing sibling static/param collisions for the same HTTP method scope.
4. Validate routes with the dedicated router conflict reference before finalizing.

## Hard Gate Behavior

If any required scaffold convention fails, the draft is non-deliverable.
Output must include `Validation Notes -> Blocking Issues` and a corrected proposal.
